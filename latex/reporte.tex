\section*{Resumen}
El proyecto consiste en una página web que toma una imagen original, detecta si tiene fondo de chroma y genera una versión limpia antes de enviarla a un chat o sistema de inteligencia artificial. La idea principal se resume así: \textit{fondo limpio y archivo optimizado}. El sistema procesa la imagen en el navegador, estima el color de chroma, permite ajustarlo con un selector y separa dos decisiones con controles independientes: chroma y comprimir. Si se activa chroma, el fondo se elimina como transparencia. Si se activa compresión, la imagen se remuestrea y se codifica para ahorrar espacio. Aunque la interfaz final es simple, el funcionamiento interno aplica conceptos de procesamiento digital de señales: adquisición de una señal visual, segmentación por color, remuestreo espacial, codificación, medición de error y modelización del ahorro.

\section{Problema de investigación}
Muchas aplicaciones modernas permiten subir imágenes a modelos de IA. El problema es que una imagen tomada con un teléfono puede pesar varios megabytes y además puede contener un fondo innecesario. En casos de chroma key, por ejemplo, el fondo verde o azul no aporta información al objeto principal y puede distraer tanto al usuario como al sistema que recibe la imagen. Enviar imágenes pesadas o con fondo artificial aumenta el tiempo de carga, el ancho de banda y el tamaño de los datos si la imagen se convierte a base64 para transmitirse por una API.

Por eso se plantea el siguiente problema de investigación: \textbf{¿cómo limpiar el fondo chroma y reducir el peso de una imagen antes de enviarla a un sistema de IA, manteniendo una versión visualmente útil para el usuario?}

Las variables principales del problema son:
\begin{itemize}
  \item Tamaño original de la imagen, medido en bytes.
  \item Tamaño de la imagen optimizada, medido en bytes.
  \item Porcentaje de ahorro.
  \item Dimensión máxima permitida después del remuestreo.
  \item Calidad JPEG usada para la compresión.
  \item Color de chroma detectado o ajustado por el usuario.
  \item Estado de los controles: quitar chroma activado/desactivado y compresión activada/desactivada.
  \item Modo de salida: fondo transparente cuando se aplica chroma.
  \item Error entre la imagen procesada y la imagen codificada.
\end{itemize}

La relevancia del problema está en que optimizar imágenes antes de subirlas reduce recursos de red y mejora la experiencia de uso. En ingeniería, esto permite tomar una decisión técnica con datos y no solo con percepción visual.

\section{Preguntas de investigación}
Para guiar el proyecto se definieron tres preguntas:
\begin{enumerate}
  \item ¿Cuánto peso se puede ahorrar al remuestrear una imagen y codificarla como JPEG?
  \item ¿Cómo detectar y comunicar el color de chroma de forma simple?
  \item ¿Qué limitaciones aparecen cuando se usa segmentación por color, transparencia PNG o compresión JPEG para preparar imágenes destinadas a sistemas de IA?
\end{enumerate}

\section{Marco teórico}
Una imagen digital puede entenderse como una señal discreta bidimensional \(x[m,n]\). Cada coordenada representa una muestra espacial y cada muestra contiene información de color. Cuando se reduce la resolución de la imagen, se reduce el número total de muestras. Este proceso se conoce como remuestreo espacial.

El prototipo también puede convertir la imagen a luminancia usando:
\[
Y = 0.299R + 0.587G + 0.114B
\]
Esta operación resume la información de color en brillo, lo cual puede ser útil cuando el color no es esencial para el análisis.

Para remover chroma, el sistema estima un color dominante en los bordes de la imagen. Después compara cada pixel contra ese color usando una distancia en el espacio RGB. Si la distancia es pequeña, el pixel se considera fondo. Si la distancia está cerca del umbral, se aplica una transición suave para evitar bordes recortados. El usuario también puede ajustar el color con un selector visual.

Cuando el control \textit{chroma} está desactivado, el sistema solo reporta el color detectado y no altera el fondo. Cuando está activado, elimina los pixeles cercanos al color seleccionado y guarda PNG para conservar transparencia. El control \textit{comprimir} es independiente: si está apagado, se conserva la resolución original y se oculta la métrica de ahorro; si está encendido, se aplica remuestreo y codificación para ahorrar espacio. JPEG usa compresión con pérdida: reduce información para obtener un archivo más pequeño. Para medir el error se calcula el error cuadrático medio:
\[
MSE = \frac{1}{N}\sum_{i=1}^{N}(x_i-\hat{x}_i)^2
\]
y la relación señal a ruido pico:
\[
PSNR = 10\log_{10}\left(\frac{255^2}{MSE}\right)
\]
En la interfaz final no se muestra PSNR para no confundir al usuario, pero sí se calcula internamente y se usa como evidencia técnica en el reporte.

\section{Herramienta moderna desarrollada}
Se creó una aplicación web con Vite, React, TypeScript y Tailwind CSS. La herramienta funciona en el navegador, por lo que la imagen se transforma antes de llegar a un servidor. Esto es importante porque el objetivo del proyecto no es solo comprimir una imagen, sino evitar que una imagen innecesariamente pesada sea enviada a una aplicación de IA.

La interfaz final se diseñó para ser directa:
\begin{itemize}
  \item El usuario sube una imagen.
  \item La página detecta el color de chroma si aparece.
  \item El usuario activa o desactiva chroma.
  \item El usuario activa o desactiva comprimir.
  \item Se muestran métricas de tamaño, ahorro de espacio y color detectado.
  \item El usuario puede descargar la salida solo cuando hay ahorro o cuando se aplicó limpieza de chroma.
\end{itemize}

Como comparación reproducible fuera del navegador, el repositorio incluye el script \texttt{scripts/remove-background.py}. Este script usa Pillow en Python para hacer chroma key por distancia de color, con los mismos dos modos conceptuales: transparencia o reemplazo por color sólido. No reemplaza al flujo web; sirve como evidencia técnica adicional de que el método también se puede ejecutar con herramientas conocidas de procesamiento de imágenes.

\section{Metodología}
La metodología aplicada fue la siguiente:
\begin{enumerate}
  \item Adquirir una imagen real desde el navegador mediante carga de archivo.
  \item Decodificar la imagen y registrar sus dimensiones y bytes originales.
  \item Estimar el color de chroma a partir de muestras en los bordes.
  \item Si el control de chroma está activo, segmentar el fondo por distancia de color y aplicar borde suave.
  \item Si el control de chroma está activo, quitar el fondo como transparencia.
  \item Si el control de compresión está activo, remuestrear la imagen para limitar su lado mayor a 1280 pixeles.
  \item Codificar la imagen resultante como PNG si conserva transparencia, JPEG con calidad 72\% si se comprime sin transparencia, o conservar la salida original cuando no hay una transformación útil.
  \item Comparar el tamaño original contra el tamaño optimizado o limpio.
  \item Calcular porcentaje de ahorro, expansión base64 evitada, MSE y PSNR.
  \item Mostrar al usuario las métricas más importantes para la decisión: original, salida, ahorro y color de chroma.
\end{enumerate}

Esta metodología define variables, aplica un método de investigación, ejecuta un experimento reproducible y permite interpretar los datos obtenidos.

\section{Predicción y modelización}
Además de medir el tamaño final, se modeló la expansión base64 con la regla:
\[
B_{64} = 4\left\lceil\frac{B}{3}\right\rceil
\]
donde \(B\) es el tamaño en bytes del archivo binario. Este modelo predice cuánto crece una imagen al serializarse para transporte textual en una API.

También se modeló el porcentaje de ahorro:
\[
Ahorro = \frac{B_{original} - B_{optimizado}}{B_{original}}\times100
\]
Con este modelo la aplicación puede informar al usuario cuántos recursos se reducen antes de subir la imagen.

Si el ahorro calculado es cero y tampoco se aplicó limpieza de chroma, la interfaz no ofrece descarga. Esto evita presentar como resultado un archivo que no mejora el original.

\section{Limitaciones técnicas}
El proyecto también reconoce límites importantes:
\begin{itemize}
  \item JPEG no conserva transparencia; una imagen PNG con canal alfa puede cambiar visualmente.
  \item La detección de chroma funciona mejor cuando el fondo es relativamente uniforme y aparece cerca de los bordes.
  \item Si el objeto tiene colores parecidos al chroma, la segmentación puede eliminar partes del objeto.
  \item El helper de Python requiere Pillow instalado; por eso se documenta como alternativa reproducible y no como dependencia obligatoria de la web.
  \item La compresión puede afectar texto pequeño o detalles finos.
  \item PSNR mide diferencia numérica, pero no garantiza que un modelo de IA interprete exactamente igual la imagen.
  \item La compatibilidad de formatos depende del navegador.
  \item El modelo de unidades de contexto es una aproximación, no el conteo interno de un proveedor específico de IA.
\end{itemize}

Para reducir estas limitaciones se podrían agregar validaciones futuras como OCR para texto pequeño, SSIM para comparación perceptual y pruebas con modelos multimodales reales.

\section{Conclusiones}
El proyecto demuestra que aplicar procesamiento digital de señales a imágenes puede resolver un problema actual: limpiar fondos innecesarios y reducir el peso de archivos antes de enviarlos a sistemas de IA. Con una configuración simple de 1280 px y calidad JPEG de 72\%, la prueba obtuvo un ahorro de 84.1\%. Además, el flujo de chroma permite quitar el fondo como transparencia, y los dos controles separan claramente la limpieza del fondo de la compresión. La interfaz final se mantuvo deliberadamente simple para que cualquier persona entienda el beneficio sin conocer MSE, PSNR, base64 o distancia RGB.

La conclusión principal es que la optimización local permite ahorrar recursos sin complicar la experiencia del usuario. El valor del proyecto está en unir una técnica de DSP con una necesidad moderna de aplicaciones web e inteligencia artificial.
